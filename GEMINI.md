# Thicket Portal API - Development Guide

## Project Overview
Thicket Portal is a Node.js backend application built with Express and TypeScript. It serves as an API for a planting/forestry management system, interacting with a MongoDB database using Mongoose. The architecture follows a layered MVC-like pattern, separating concerns into models, controllers, services, routes, and validation.

**Key Technologies:**
- **Runtime & Framework:** Node.js, Express
- **Language:** TypeScript (configured for `nodenext` ESM modules)
- **Database:** MongoDB with Mongoose
- **Authentication:** JSON Web Tokens (JWT) and bcrypt
- **Validation:** Joi

## Directory Structure
- `src/` - Application source code.
  - `config/` - Database connection and configuration setup.
  - `controllers/` - Express route handlers mapping HTTP requests to services.
  - `errors/` - Custom application error classes (e.g., `AppError`).
  - `middleware/` - Express middleware for authentication, error handling, etc.
  - `models/` - Mongoose schemas and database models (e.g., Job, Planter, Manager).
  - `routes/` - Express router definitions.
  - `services/` - Business logic and database operations.
  - `types/` - TypeScript type definitions and interfaces.
  - `utils/` - Shared utility functions (e.g., JWT helpers).
  - `validation/` - Joi schemas and validator functions.
- `dist/` - Compiled JavaScript output (generated by `tsc`).

## Building and Running
The project uses `tsx` for running TypeScript directly during development.

**Development Server:**
```bash
npm run dev
```
This runs `tsx watch src/server.ts` to start a development server with hot-reloading.

## Development Conventions
1. **Module System:** The project uses ECMAScript Modules (ESM). Ensure imports include the `.js` extension (e.g., `import { connectDB } from "./config/db.js"`), as configured by `"moduleResolution": "nodenext"` in `tsconfig.json`.
2. **Layered Architecture:** Keep business logic in `services/`, request handling in `controllers/`, and routing in `routes/`.
3. **Validation:** All incoming request payloads should be validated using Joi schemas defined in `src/validation/`.
4. **Error Handling:** Use custom error classes (like `AppError`) and pass errors to the `next()` function to be caught by the global error handling middleware (`src/middleware/errorMiddleware.ts`).
5. **Authentication:** Endpoints requiring auth use middleware (e.g., `authMiddleware`) which leverages JWTs for verifying users (Planters, Managers, Team Leads).
